# NodeTorch Codebase Survey — 2026-06-21

Whole-repo, read-only survey across six dimensions (refactoring, organization,
testing, unfinished work, documentation, next steps). ~31.5k LOC.

**Headline:** the architecture is sound — clean 6-layer separation, registered
execution modes, a node-builder registry, a training-paradigm plugin system, and
a recently-added agent. The **backend is in good structural shape** (thin
`main.py`, `api/` split by concern, decomposed `engine/graph_builder/`). The
debt is concentrated in three places, each of which surfaced independently in
multiple dimensions:

1. **The frontend has no API/transport layer** — 35 hardcoded `localhost:8000`
   URLs across 10 files; flagged by refactoring, organization, *and* next-steps.
2. **`src/ui/useGraph.ts` (1435 lines) is a god-file** — flagged by refactoring,
   organization, and testing (it's also the single largest untested surface).
3. **Backward step-through is a fully-wired, compute-heavy feature whose result
   is thrown away** behind a "coming soon" placeholder.

## 1. Refactoring

### High impact
- **The per-node-type forward-dispatch loop is copy-pasted 4–5×.** The
  gather-inputs → branch-on-node-type → pack-metadata chain is re-implemented,
  with drift, in `backend/engine/graph_builder/forward.py:28`
  (`build_and_run_graph`), `inference.py:201` (`infer_graph`), `inference.py:24`
  (`evaluate_test_set`), `inference.py:459` (`_probe_tracked_samples`), and
  `backend/training/base.py:546` (`run_final_forward`). They already diverge —
  the diffusion/timestep handling in `forward.py:155` is *missing* from
  `run_final_forward`. **Extract one `run_module_for_node(node, modules, inputs,
  ctx)` dispatcher** shared by all callers. Biggest correctness-risk dedup in the repo.
- **The torchvision dataset family is 4× copy-pasted** in
  `backend/dataprep/data_loaders.py` — `load_*` (`:29/:111/:151/:62`),
  `train_dataset_*` (`:44/:80/:125/:165`, cifar10/cifar100 aug blocks byte-identical),
  `denormalize_*` (`:55/:102/:143/:175`), `detail_*` (`:591/:620/:675/:700`).
  Collapses (~250 lines) to a `TorchvisionDatasetConfig` dataclass + 4 generic
  functions. Biggest line-count win.
- **Node-type sets duplicated frontend↔backend and re-listed inline.** Canonical
  sets in `backend/engine/graph_builder/constants.py:10` are re-declared in
  `src/core/validation.ts:64-65` (`LOSS_TYPES`/`OPTIMIZER_TYPES` — already
  structurally different) and re-spelled inline at `backend/training/__init__.py:34`,
  `backend/visualize/loss_landscape.py:58`, `src/ui/step-through/StepThroughPanel.tsx:161`,
  `backend/export/model.py` (multiple). Single-source per side.
- **`useGraph.ts` repeats the fetch→catch→json→status block per endpoint** —
  `runForward` (`:635`), `runInfer` (`:704`), test/backprop/export/weights
  (`:784/:853/:889/:936/:990`) each repeat the same 4-stage boilerplate (the
  `'Cannot connect to backend'` string appears 8×). Wrap in one
  `callBackend(endpoint, graphData)`.

### Medium
- **Optimizer/scheduler knowledge mirrored with drifting defaults.**
  `backend/training/base.py:319/:343` (`build_optimizer`/`build_scheduler`) vs
  `backend/export/training_loops.py:428/:449`. Defaults already diverge: lr
  `0.01` vs `0.001`; StepLR `step_size=10,gamma=0.5` vs `epochs//3,gamma=0.1`.
  **Exported scripts therefore train differently from the app.** Single config source.
- **Conv/pool output-shape formula** `floor((H+2P-K)/S)+1` hand-written in
  `conv2d.ts:78`, `conv1d.ts:31`, `maxpool2d.ts:51`, `avgpool2d.ts:30`,
  `conv-transpose2d.ts:31`. Extract `convOutputSize(...)` in a domain shape-math module.
- **Leaky layering:** ML type strings in Layer 2/3 — `validation.ts:64-65`,
  `engine.ts:130/:152` (`'subgraph.input'/'subgraph.output'` literal checks),
  `engine.ts:157-180` (domain-specific `paramCount`/`paramBreakdown` formatting in
  Layer 3). Express via NodeDefinition capability flags instead of literal comparisons.

## 2. Organization

The frontend carries the structural debt; the backend is largely well-factored.

- **Split `src/ui/useGraph.ts` (1435 lines)** — ≥5 concerns in one hook:
  serialization (`:89-145`, pure, belongs in `src/core/serialization.ts`),
  backend comms (`:633-1192`, ~560 lines → `src/ui/api/`), undo/redo (`:175-378`),
  saved blocks (`:203-222`), subgraph navigation (`:223-342`), auto-organize
  layout (`:1193-1267`, pure graph algo → `core`), copy/paste (`:1268+`).
- **Introduce a frontend `src/ui/api/` layer** mirroring `backend/api/` — there
  are **35** hardcoded `localhost:8000` literals across `App.tsx`, `Toolbar.tsx`,
  `StepThroughPanel.tsx`, `DatasetDetail.tsx`, `AgentSettings.tsx`, `useGraph.ts`,
  `TokenizerDetail.tsx`, `useAgentChat.ts`, `LayerDetail.tsx`, `TrainingDashboard.tsx`.
  This is the single biggest asymmetry between the two halves.
- **Promote canvas/viz primitives to a shared `src/ui/canvas/` module.**
  `step-through/transformations/shared.tsx` already has the right primitives but
  is buried 3 levels deep, so others reimplement: `heatColor` defined 2× (`AttentionViz.tsx:570`,
  `LayerDetail.tsx:453`), a `Heatmap` component 3×, `displayToken` 3×, and raw
  `getContext('2d')` pixel rendering hand-rolled in **16 files**.
- **Split `TrainingDashboard.tsx` (1285 lines)** — ~15 components already marked
  by `// --- X panel ---` dividers; charts (`Chart:553`, `GradientFlowChart:786`,
  `PerClassChart:939`) → `dashboard/charts/`, interfaces (`:6-89`) → `dashboard/types.ts`.
- **Split `LayerDetail.tsx` (981 lines)** — viz modes → `inspector/detail-views/`,
  canvas widgets → the shared `canvas/` module above.
- **Backend smaller splits:** `visualize/layers/misc.py` (409 lines) hides
  attention/recurrent/structural families → split per the "one file per family"
  convention; `dataprep/data_loaders.py` mixes loaders + Dataset classes +
  detail-info + augmentation; `training/base.py` → extract `training/metrics.py`
  + `training/optim.py`; `graph_builder/inference.py` (658) → split test-eval +
  tracked-samples out.

## 3. Testing

**Well-tested:** `core/graph.ts`, `core/validation.ts`, shape executors,
`chat/graphTools.ts`, `backend/agent/`, `node_builders.py`, `export/`, tokenizer.
**Untested & load-bearing:** `useGraph.ts` (0 tests), `core/engine.ts` (0),
`backend/engine/graph_builder/` build→forward→inference (0), `training/` loop
execution (only helpers), the WebSocket transport on both sides.

Highest-risk gaps:
1. **Serialize/deserialize round-trip** (`useGraph.ts:101-141`, incl. nested
   subgraphs) — the persistence + undo + agent-bridge + backend-payload format,
   completely untested. Functions aren't exported — needs a small refactor to
   test. *Single most load-bearing untested code in the repo.*
2. **Undo/redo + batching** (`useGraph.ts:183-201`) — one-undo-per-batch, nested
   `beginBatch` no-op, 50-entry cap, redo-clear-on-mutation. Off-by-one here
   silently corrupts history.
3. **WS training message handler** (`useGraph.ts:1064-1137`) — mock-WebSocket
   test feeding `epoch×N → done`, asserting `trainingProgress` shape (which
   `graphTools.test.ts:174` already depends on but nothing produces under test).
4. **`graph_builder` build→forward integration** — feed a minimal serialized
   graph to `execute_graph` (`forward.py:393`) and a subgraph to
   `build_subgraph_module` (`build.py:222`).
5. **`engine.ts` execution + subgraph recursion** (`engine.ts:97-210`) — order,
   dirty-skip, multi-output port routing, subgraph param aggregation.
6. **`run_training` never actually runs** — `test_training.py:4` imports it but
   never calls it. Add 1-epoch smoke tests per paradigm + `detect_training_mode`
   cases for gan/diffusion/autoregressive.
7. **Preset → `execute_graph` round-trip** — guards the TS↔Python `SerializedGraph`
   contract from drift (extend the `test_export.py:158` preset pattern).
8. **`friendlyError` table test** (`useGraph.ts:146-158`) — user-facing educational
   copy via brittle regex; a PyTorch version bump silently degrades it.

**Brittle existing tests:** `test_api.py:84` (`test_route_count_is_41` magic
number, redundant with the exact-set test) and the exact tool-name set /
call-order sequences in `test_agent.py:95,304` (intentional but ordering-coupled).

  discarded. `get_backward_viz` (`node_viz.py:112`) unconditionally `return {}`.

- **Dead endpoints `/save-model`, `/load-model`, `/saved-models`**

- **Minor dead code:** `insights.ts:20` (`fallbackInsight`, unimported),

## 6. Next Steps (highest leverage)

- **Backend hot reload** — `main.py:52` runs uvicorn with no `reload`; CLAUDE.md
  calls manual restart the top friction. `uvicorn.run("main:app", reload=True)`
  gated behind `NODETORCH_DEV=1`. One line, large daily payoff.
- **Validate the serialization contract on load** — `deserializeGraph`
  (`useGraph.ts:139`)/`loadGraph` (`:567`) never check `version` or that node
  types still exist; a shared/old/hand-edited graph crashes mid-load via
  `createNode`/`addGraphEdge` throws. Add `validateSerializedGraph()` returning
  friendly errors. Most likely thing to bite students sharing graphs.
- **Persist the trained-model store** — `_model_store` is in-memory, keyed
  `"current"` (`_state.py:36`); a restart/hot-reload loses minutes of training
  with no recovery. Auto-snapshot `current.pt` at end of `train_graph`, lazy-load
  on first inference. Closes the worst data-loss hole in the core loop.
- **Centralize the backend URL** (see §2) — also unblocks remote/GPU backends and
  hosted demos via `import.meta.env.VITE_API_BASE`.
- **Honor dirty tracking + debounce in `runShape`** — `useGraph.ts:330` re-executes
  the *whole* graph on every edit from ~15 call sites, ignoring the `markDirty`
  machinery `graph.ts:201-217` already maintains. Hot path for the headline
  "instant shape inference" feature on 30+ node presets.
- **Guard concurrent training** — no lock around the global `_model_store`; two
  tabs corrupt results silently. Reject/queue in the train path.
- **Backend-down UX** — failures are uniformly `catch(() => {})` (`useGraph.ts:217/600/610/628`,
  `TrainingDashboard.tsx:122+`); a student who forgot to start the backend sees
  nothing. One shared "Backend offline" banner.
- **Node-anchored validation hints (educational)** — `ValidationError` already
  carries `nodeId` (`validation.ts:16`) and `EngineNode.tsx` already renders
  `error` metadata; surface each error as a badge on the offending node. Mostly
  UI wiring over existing data; squarely on-mission.

   repeated fetch boilerplate, and unblocks non-localhost use (one refactor, three findings).

   forward-dispatch copies (correctness risk).

   no-op pipeline; remove its dead types/registry.
7. **Fix `CONTRIBUTING.md` backend file map** + write `docs/agent.md` — onboarding
   is currently broken and the largest subsystem is undocumented.
8. **Add the backend execution-path integration tests** (`graph_builder` build→forward,
   `run_training` 1-epoch smoke per paradigm, preset→`execute_graph`).
